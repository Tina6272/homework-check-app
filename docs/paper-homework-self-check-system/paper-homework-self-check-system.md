# 紙本作業自主訂正系統

## 1. 核心目標

作業檢查 App 的第一階段，不是要取代老師批改，也不是一開始就用 AI 判題。核心目標是讓學生在紙本作業完成後，照著清楚流程自己檢查、標記不確定題目、完成訂正，最後把狀態回報給老師與家長。

產品方向：學生自主完成作業。

第一階段要避免：

- 一開始就做照片上傳、OCR、AI 判題。
- 讓家長變成主要管理者。
- 儲存學生姓名或敏感個資。
- 在資料流程未定案前就急著接資料庫。

## 2. 三端角色

### 學生端

學生端是主流程，應該讓學生能完成：

1. 輸入班級代號與座號。
2. 選擇今天要完成的作業。
3. 回報紙本作業已完成。
4. 依自我檢查清單逐項確認。
5. 標記不確定、已訂正、仍卡住的題號。
6. 送出目前狀態。

學生端文字要短、清楚、可操作。按鈕要引導學生下一步，而不是只呈現統計。

### 老師端

老師端用來派作業與看風險，不是逐題批改系統。第一階段應優先提供：

- 派發 A/B/C 班作業。
- 查看每班完成、待訂正、卡住、待老師確認的人數。
- 查看風險訊號，例如完成太快、自我檢查不足、仍有卡住題目。
- 對提交狀態做簡單確認或標記需要教學。

老師端可以延後：

- 複雜成績表。
- 大量逐題批改。
- OCR 與 AI 判題管理。

### 家長端

家長端只做提醒與進度查看，避免變成家長代替學生管理。第一階段可顯示：

- 今天是否有作業。
- 學生目前狀態：未開始、進行中、待訂正、卡住、完成。
- 是否需要家長提醒。
- 獎勵點數或連續完成次數。

家長端不應讓家長直接幫學生改成完成，也不應顯示過多老師端風險操作。

## 3. 建議狀態名稱

```text
notStarted      尚未開始
inProgress      作業進行中
selfCheck       自我檢查中
needsCorrection 需要訂正
peerCheck       同儕檢查中
teacherReview   等老師確認
done            已完成
stuck           卡住，需要協助
```

狀態命名要先和介面文字一起討論定案，再接 Firestore。

## 4. 第一階段資料流

```text
老師派作業
  -> 學生選作業
  -> 學生回報紙本完成
  -> 學生自我檢查
  -> 學生標記不確定與已訂正題號
  -> App 產生提交狀態與風險訊號
  -> 老師查看需要協助或抽查的學生
  -> 家長查看進度與提醒
```

第一階段使用 Firestore-only：

- 作業資料
- 學生座號統計
- 提交狀態
- 自我檢查項目
- 不確定、已訂正、卡住題號
- 風險標記

第一階段不使用：

- Storage 照片
- OCR
- AI 判題
- 學生姓名

## 5. Firestore 草案

詳細欄位以 `docs/firebase-data-model.md` 為準。主要結構：

```text
classes/{classCode}
  assignments/{assignmentId}
  students/{studentSeat}
  submissions/{submissionId}
    questionChecks/{questionId}

assignmentTemplates/{templateId}
```

## 6. 下一輪討論順序

1. 打開目前前端原型。
2. 先看學生端是否真的能支援「學生自主完成作業」。
3. 調整學生端文字、按鈕與流程順序。
4. 再看老師端需要先看到哪些資訊。
5. 最後看家長端是否過度介入。
6. 三端流程與狀態名稱定案後，再接 Firestore。

## 7. 後續階段

第二階段可以加入答案庫與逐題檢查，但仍應以學生自主訂正為核心。

第三階段才評估照片上傳、OCR、AI 判題。這些功能成本較高，也會增加資料安全與操作複雜度，不應放在第一階段。
